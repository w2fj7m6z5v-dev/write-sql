>
> 用途：先做案例导航和处理顺序，不直接沉淀到 `write-query`。每个案例仍需按 `sql-case-sedimentation` 的“一案一确认卡”逐条确认。

## 使用方式

- 先看“推荐处理顺序”，选一个案例进入逐案分析。
- 点“源文件”查看原 SQL / TXT。
- “初步场景”和“关键证据”只用于排队和定位，不代表最终沉淀结论。
- “沉淀等级”默认保留为待确认；只有逐案分析后才改为“直接沉淀 / 待核对 / 不沉淀 / 已覆盖不写入”。

## 推荐处理顺序

1. **先处理已知路由或明显覆盖的轻量案例**：地址层级、状态回填、固话/宽带资料清单。这类案例适合校准“已覆盖不写入”和小范围补规则。
2. **再处理销售品、特性、设备、状态类补表案例**：常见沉淀点是 `FIELD_BACKFILL.md`、`RULES.md` 或表文档的补表链路。
3. **再处理收入、积分、专线、网格汇总类案例**：这些更容易涉及指标口径、专项流程或多步 CTAS。
4. **最后处理多表复杂专项**：如责任田收入、单移 3C 认领、校园 VPN 大数据打标；先拆出可复用流程，再判断是否值得进 `scenarios/`。
