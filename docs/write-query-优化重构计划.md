# write-query 系统审计与重构计划

## 目标

对 `write-query` 做一次证据化系统审计，检查逻辑正确性、知识重复与冲突、运行流程、维护成本和质量保障，再决定哪些部分值得重构。重构不是预设结论；本轮先完成审计报告和重构候选，不在未确认前改变运行时行为。

## 已确认的决策

- 现有 CTAS、禁用 `WITH`、确认流程和精简输出等规则全部可以质疑，但必须区分业务硬约束、设计选择和历史遗留。
- 正确性以仓库内表文档、指标、场景、案例、脚本和 CI 为依据；无法由仓库证明的生产事实标记为待外部核验。
- `.agents/skills/write-query` 作为本轮审计工作树，`.claude/skills/write-query` 作为镜像 Adapter；双入口不计为知识重复，但必须检查镜像漂移。
- 审计期间保留用户已有修改，不暂存、不清理、不回退，不触碰无关文件。
- 审计完成后设置决策闸门；用户确认候选后，才另行实施重构。

## 审计范围

包括 `SKILL.md`、索引、路由、字段补表、规则、表文档、指标文档、场景、验证案例、字典、校验脚本、双入口同步脚本和 PR CI。

不把 `D_experience/_archive` 当作运行时资产，但会检查运行时文件是否错误引用归档内容。仓库中不存在 `CONTEXT.md` 或 ADR，本轮将记录这一事实，不假定不存在的领域决策。

## 执行阶段

### 1. 基线固化

- 记录 Git 状态、文件数量、行数、双入口比较结果和现有检查退出码。
- 生成运行时文件清单、索引登记关系和 Markdown 引用关系。
- 比较双入口时排除 `.DS_Store` 和既定 `*-workspace`；其它差异全部保留为审计证据。

### 2. 运行流程审计

还原标准指标、普通路由、多指标、字段补表、专项场景、验证案例、直接生成和低置信度确认路径。

对每个 Module 检查 Interface、Implementation、Depth、Locality 和 Leverage，并使用删除测试判断它是否真正集中复杂度，还是只是把复杂度转移到调用方。

### 3. 知识归属审计

建立事实归属矩阵，检查以下权威关系：流程归 `SKILL.md`，路由归 `ROUTING.md`，表元数据归表文档，标准指标归指标文件，通用补表归 `FIELD_BACKFILL.md`，通用 SQL 规则归 `RULES.md`，专项链路归 `SC-*.md`，验证实例归 `VC-*.md`，维护规范归 `MAINTENANCE.md`。

重复内容分为必要摘要、有意镜像、危险副本、真实冲突和疑似陈旧，不把所有重复文本都直接判为问题。

### 4. 完整性与质量保障审计

- 对账表、指标、场景和案例索引，检查重复 ID、孤儿文件、断链、旧路径、废弃表名和数量漂移。
- 对比 `RULES.md` 与 `audit_sql.py` 的规则覆盖，检查脚本退出码、警告策略、版本契约和测试缺口。
- 检查 `lint_metric_index.py` 是否覆盖全部指标表格、重复 ID、畸形行、孤儿文件和未登记文件。
- 检查 CI 对删除文件、带空格路径、Markdown 内 SQL、全库检查和同步噪声的覆盖。

### 5. 回归样本与报告

建立至少 12 类回归样本，覆盖单指标、多指标、主表路由、低置信度、补表、JOIN 炸行、专项场景、附件种子、日/月表、码值、敏感字段和直接给 SQL 等路径。

最终报告为 `docs/write-query-系统审计报告.md`。每个问题使用 `WQ-###` 编号，记录类别、严重度、置信度、证据、影响、建议、关联回归样本和是否需要外部核验。

## 当前已知待复核线索

- 多指标 B 类策略出现“拆 CTE”，但正式交付规则又禁止 `WITH`。
- 主宽入网条件同时出现在指标文件、069 表文档、`RULES.md` 和 `SKILL.md`。
- `TABLE_INDEX` 的实际内容超出其索引职责，包含路由和专项说明。
- `MAINTENANCE.md` 与 `ROUTING.md` 对专项唯一索引位置的描述存在漂移。
- 指标 lint 只解析首个表格，且表文档缺失只产生警告。
- 表文档 frontmatter 覆盖率不足，VC 索引宣称的完整 SQL 与实际案例内容可能不一致。
- SQL 静态审计覆盖范围小于 `RULES.md`，且缺少单元测试和全库回归。
- 本地双入口检查会被 `.DS_Store` 触发假失败，但排除后内容可能一致。

以上只作为审计入口，必须重新验证后才能定性。

## 问题分级

- P0：高概率生成错误 SQL、敏感数据暴露或严重结果错误。
- P1：关键路径失效、规则冲突或常见需求结果不稳定。
- P2：重复维护、检查盲区、流程冗余或较高维护成本。
- P3：命名、文档漂移和低风险清理问题。

## 决策闸门

审计完成后暂停运行时修改，由用户按问题编号和重构候选确认：哪些规则是生产硬约束、哪些问题立即修复、哪些 Module 值得深化、是否接受行为变化。闸门通过后再建立 characterization tests，随后按低风险纠错、知识结构调整、流程行为调整、CI 与双入口治理的顺序实施。

## 验收标准

- 运行时文件全部完成职责分类，表、指标、场景和案例索引全部完成对账。
- 所有已知线索都有确认、否定或待外部核验结论。
- 每个确认问题都有文件和行级证据。
- 每个危险重复都有唯一事实源或明确保留理由。
- 至少完成 12 类回归样本及预期不变量。
- 审计报告给出排序后的 3 至 5 个重构候选。
- 审计期间除计划和审计报告外不修改运行时技能文件。
