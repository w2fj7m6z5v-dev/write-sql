# write-query 系统审计报告

## 审计结论

当前 `write-query` 已经形成了较完整的运行路径，但整体质量保障仍是“人工语义审视 + 局部静态检查”。主要问题不是缺少资料，而是部分 Module 的职责、事实归属和检查覆盖没有完全对齐，导致维护者需要在多个文件之间来回确认。

本轮确认了 9 个问题：2 个 P1，7 个 P2；另有 4 个可深化方向。未对业务生产数据做外部核验，以下结论仅基于仓库证据。

## 审计范围与基线

- 运行树：`.agents/skills/write-query`。
- 镜像树：`.claude/skills/write-query`。
- 表文档：128 份；指标文档：92 份；场景：11 份；验证案例：16 份。
- 表文档中具备 `table_id` frontmatter 的有 38 份。
- 16 个验证案例中有 SQL 代码块的有 8 个。
- `lint_metric_index.py` 两个入口均返回通过，报告 91 行指标、无断链。
- `diff -qr -x '*-workspace' -x '.DS_Store' .agents/skills .claude/skills` 返回一致。
- `bash scripts/sync_skills.sh check` 返回失败，原因仅为 3 个 `.DS_Store` 差异。
- 当前工作区存在用户已有修改，本轮只新增计划和审计报告，没有处理其它修改。

## 运行流程

```text
需求
  -> SKILL.md 路径判定
  -> METRIC_INDEX / ROUTING
  -> metric / table
  -> FIELD_BACKFILL / scenarios
  -> RULES / dictionaries
  -> SQL + 口径 + 风险 + 自检
```

该流程的主干是清楚的：标准指标优先走指标路径，普通需求走路由路径，字段缺口进入补表，复杂专项进入 SC，具体实例进入 VC。主要摩擦集中在职责声明与实际内容不一致、权威规则被多处重复以及验证工具覆盖不足。

## 问题清单

### WQ-001：多指标 B 类允许 CTE 与正式禁用 WITH 发生冲突

- 严重度：P1
- 置信度：已确认的规则文本冲突
- 状态：已修复，待回归验证
- 修复前证据：`SKILL.md:50` 写明“拆 CTE 后再汇总”；`SKILL.md:238` 和 `SKILL.md:252` 又规定正式交付禁止 `WITH`。
- 修复内容：B 类现在要求先判断能否共用一次扫描；能共用时各指标条件独立放入 `CASE WHEN`，不能共用时拆多步 CTAS，并明确不同指标的时间条件不得合并进公共 `WHERE`。
- 影响：模型在同表异轴多指标场景中无法确定 CTE 是允许的内部编排、非正式示例，还是应改成多步 CTAS。该冲突可能影响 SQL 形态选择。
- 建议：将“逻辑拆分”和“SQL 交付形态”分开定义，明确 B 类最终只能采用何种 CTAS 编排，并增加正反例回归。

### WQ-002：运行时文档存在已失效的归档规则链接

- 严重度：P1
- 置信度：已确认
- 状态：已修复，待回归验证
- 证据：`tables/041_优惠订单表.md:22`、`:46` 引用 `../D_experience/cdap_global_rules.md`；该路径不存在，实际文件位于 `D_experience/_archive/cdap_global_rules.md`。
- 修复内容：双入口 041 表文档已改为引用当前 `RULES.md`，并明确 `RULES.md` 是规则权威；未把 `_archive` 恢复为运行时入口。
- 影响：读取 041 表文档时会进入断链；维护者无法判断这是应恢复的运行时规则、应改为新规则指针，还是仅保留历史来源。
- 建议：先确认该引用对应的规则是否已完整迁移，再删除旧引用或改为当前权威文件指针；不能直接把 `_archive` 重新变成运行时入口。

### WQ-003：标准指标事实在多个 Module 中重复维护

- 严重度：P2
- 置信度：已确认存在重复，是否造成结果差异需继续核对
- 状态：已修复，待回归验证
- 证据：主宽入网条件分别存在于指标文件、069 表文档的常用条件、`RULES.md` 的通用规则和 `SKILL.md` 的运行流程说明中。代表性证据为 `metrics/基本面/M-BASIC-BB-001_主宽入网数.md:27`、`tables/069_全业务资料表.md:258`、`RULES.md:51`。
- 修复内容：完整指标公式收敛到单指标文件；069 表文档、`RULES.md` 和双入口 `SKILL.md` 改为保留用途说明、加载路径和通用审计规则，不再复制完整筛选条件。
- 影响：修改指标条件时需要判断哪些位置必须同步，重复内容容易出现“指标文件已更新但规则摘要未更新”的漂移。
- 建议：为指标公式、表元数据、通用审计和流程说明划定唯一事实源；其它位置只保留用途说明和指针。保留必要摘要，但不保留完整条件副本。

### WQ-004：TABLE_INDEX 的职责声明与实际内容不一致

- 严重度：P2
- 置信度：已确认
- 证据：`TABLE_INDEX.md:3`、`:10`、`:15` 声明只做表名/Hive 名/路径查找，但索引列包含 `grain`、`partition_keys`、`use_when`、`avoid_when`，并在 118 表条目中写入 2021 年重建路径和专项选择建议。
- 影响：表索引、ROUTING、表文档三者都可能成为路由规则的候选事实源，维护者难以判断应改哪一层；读取路径也会发生重复跳转。
- 建议：审计每一列的实际使用者，决定保留最小表目录还是明确承认它也是轻量路由索引。不能只改标题，必须同步调整 `ROUTING` 和维护准则。

### WQ-005：维护准则与路由索引位置发生漂移

- 严重度：P2
- 置信度：已确认
- 证据：`MAINTENANCE.md:48` 说 `ROUTING.md` 的专项场景索引是唯一表；`ROUTING.md:30` 已明确专项场景唯一索引是 `scenarios/INDEX.md`。
- 影响：新增 SC 时维护者可能错误修改 ROUTING，造成索引重复或专项规则回流核心文件。
- 建议：保留 `scenarios/INDEX.md` 作为唯一场景目录，并把维护准则、ROUTING 和 `scenarios/INDEX.md` 的表述统一为同一套规则。

### WQ-006：指标 lint 只覆盖首个表格，且未检查全量结构

- 严重度：P2
- 置信度：已确认
- 证据：`lint_metric_index.py:29-33` 找到第一个 `metric_id` 表头后即停止搜索；`METRIC_INDEX.md:124-130` 另有“分类口径”表。实际指标文件 92 份，而脚本报告 91 行。
- 影响：分类指标、重复 ID、畸形行、孤儿指标文件和未登记文件可能不会进入校验；当前“通过”不能代表整个指标目录完整。
- 建议：把索引解析从“首个表格”提升为完整文档检查，至少覆盖所有表格、重复 ID、断链、孤儿文件和失败关闭策略。

### WQ-007：表文档结构化元数据覆盖不足

- 严重度：P2
- 置信度：已确认覆盖率，不判定缺失内容必然错误
- 证据：128 份表文档中仅 38 份包含 `table_id` frontmatter；`TABLE_INDEX.md:3-10` 又要求落 SQL 前核对 frontmatter、字段、分区和粒度。
- 影响：同一表的关键元数据在不同文档格式中表达，自动检查无法稳定判断表 ID、生产名、粒度和分区是否存在。
- 建议：按字段可证实程度设计结构化元数据迁移，不得凭推测补生产信息；无法确认的字段要显式标记未知，而不是伪造完整 frontmatter。

### WQ-008：VC 的定义与实际内容不稳定，无法直接作为回归集

- 严重度：P2
- 置信度：已确认
- 证据：`verified-cases/INDEX.md:3-5` 将 VC 定义为已验证具体实例并称含完整 SQL；16 个 VC 中只有 8 个包含 SQL 代码块，`VC-20260608-001` 主要是流程、口径和字段说明。
- 影响：部分 VC 可作为 SQL golden fixture，部分只能作为知识案例；当前统一加载或统一回归会产生错误期待。
- 建议：将案例显式分类为 `golden SQL`、`knowledge case` 或待补齐案例，并为可回归案例补输入、预期不变量、审计期望和验证信息。

### WQ-009：CI 和本地检查存在覆盖盲区

- 严重度：P2
- 置信度：已确认工具链行为
- 证据：`audit_sql.py:251-260` 只有 7 条注册规则，而 `RULES.md` 还要求检查 `WITH`、裸 `SELECT`、ORC/Snappy、英文 snake_case、字段存在性、粒度和 JOIN 放大等内容。`.github/workflows/pr-review.yml:25` 使用 `--diff-filter=ACMRT`，`:34-46` 通过空格分隔 SQL 路径，`:54-61` 只在 METRIC_INDEX 变更时 lint，`:63-71` 又重复同步检查。
- 影响：静态审计覆盖不了大量运行时约束；删除文件、带空格路径、Markdown 内 SQL、未直接修改索引但影响指标的变更可能漏检。
- 建议：先建立规则—实现—测试矩阵，再补测试和统一校验入口；保留现有 CLI 兼容性，避免一次性替换导致 CI 行为不可追踪。

## 非问题确认

- `.agents` 与 `.claude` 在排除 `.DS_Store` 后内容一致，因此双入口复制本身不是知识重复问题。
- 两套 `METRIC_INDEX.md` 的现有路径检查均通过，当前没有发现已登记指标文件或表文件断链。
- `.DS_Store` 已被 `.gitignore` 忽略，当前同步失败属于本地比较脚本的噪声处理问题，不是技能内容漂移。
- 本轮没有对任何指标、表字段或生产业务口径作外部正确性结论。

## Deepening Opportunities

### 1. 统一知识登记与引用校验 Module

- Files：指标、表、场景、案例索引与现有 lint 脚本。
- Problem：索引解析不完整，职责和登记关系分散，检查通过不能代表目录完整。
- Solution：把文件登记、路径、ID、类型、状态和引用关系集中到一个可验证的知识登记 Module；保留各类索引作为面向运行时的轻量视图。
- Benefits：提高 Leverage，降低维护者在多个索引间切换的成本；Interface 变小后，测试可以覆盖完整登记关系而不是只测单个表格。

### 2. 统一规则事实源与运行编排 Module

- Files：`SKILL.md`、`ROUTING.md`、`TABLE_INDEX.md`、`FIELD_BACKFILL.md`、`RULES.md`、指标和表文档。
- Problem：流程、指标和表附注之间有重复事实，规则优先级虽然存在但不能消除所有副本。
- Solution：按事实类型重新划分 Module，核心文件只保留流程和指针，复杂业务规则下沉到唯一权威文件。
- Benefits：提升 Locality，减少一次口径变更的编辑范围；回归测试可以针对每个事实源和引用者分别验证。

### 3. 将 VC/SC 变成可分层验证的案例 Module

- Files：`scenarios/`、`verified-cases/`、SQL 审计脚本和新增回归清单。
- Problem：可复用规则、完整 SQL、知识说明和验证证据混在同一案例类型中。
- Solution：明确专项规则、知识案例和可执行 golden SQL 的职责与最低字段，按类别建立不同验收方式。
- Benefits：案例 Interface 更清楚，测试可以验证 SQL 不变量、必读资料和风险触发，而不要求所有案例都逐字匹配。

### 4. 建立统一质量门与双入口 Adapter 治理

- Files：`audit_sql.py`、`lint_metric_index.py`、`sync_skills.sh`、PowerShell 版本和 PR CI。
- Problem：脚本检查范围不一致，镜像检查重复，`.DS_Store` 等环境噪声会产生假失败。
- Solution：统一排除语义、退出码和报告格式，增加规则正反例、索引完整性和双入口一致性测试。
- Benefits：把质量控制从人工记忆变成可重复的质量门；两个入口继续保留，但 Seam 更清楚，修改传播路径更可见。

## 推荐优先级

1. 先处理 WQ-001、WQ-002、WQ-005 的规则冲突和失效引用。
2. 再处理 WQ-006、WQ-008、WQ-009，建立可验证的回归和质量门。
3. 最后评估 WQ-003、WQ-004、WQ-007 的知识结构深化，避免在缺少回归保护时大规模搬迁内容。

## 决策闸门

本报告完成了第一轮审计，但没有自动修改运行时技能。下一步需要按 `WQ-001` 至 `WQ-009` 和 4 个深化候选确认优先级；确认后先补 characterization tests，再进入对应 Module 的具体 Interface 设计和重构实施。
