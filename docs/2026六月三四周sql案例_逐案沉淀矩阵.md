> 用途：先做案例导航和处理顺序，不直接沉淀到 `write-query`。每个案例仍需按 `sql-case-sedimentation` 的"一案一确认卡"逐条确认。

## 使用方式

- 先看"推荐处理顺序"，选一个案例进入逐案分析。
- 点"源文件"查看原 SQL / TXT。
- "初步场景"和"关键证据"只用于排队和定位，不代表最终沉淀结论。
- "沉淀等级"默认保留为待确认；只有逐案分析后才改为"直接沉淀 / 待核对 / 不沉淀 / 已覆盖不写入"。

## 推荐处理顺序

1. **先处理明显已覆盖的轻量案例**：号码状态回填、宽带到达统计、网格收入统计、主套餐查询、入网信息回填。这类案例适合校准"已覆盖不写入"。
2. **再处理补表/补字段类案例**：地址层级下钻、订单信息补表、客户信息匹配、专线字段补全。常见沉淀点是 `FIELD_BACKFILL.md`、`ROUTING.md`。
3. **再处理积分、收入、离网、降档类口径案例**：这些更容易涉及指标口径、审计规则或专项流程。
4. **最后处理多表复杂专项**：如政企汇工作产品匹配、百度巡检清单、欠费订单匹配、VPN 群优惠到期等；先拆出可复用流程，再判断是否值得进 `scenarios/`。
