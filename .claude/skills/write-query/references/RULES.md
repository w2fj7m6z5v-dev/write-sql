# SQL Generation + Review 规则

用于 SQL 生成后的**通用** SQL 审计与编排。只加载与本次 SQL 相关的小节；码值仍按需读取 `D_experience/dictionaries/{field}.md`。专项场景审计见 `scenarios/SC-*.md`，本文件不重复专项长规则。

## 使用方式

SQL 写完后先做“最小审视闭环”，再按风险触发补充审视。审视是内部质量动作，默认不输出完整审计表；只要发现会影响结果或执行的“风险”，交付时必须写进待确认项，不要把风险藏在解释里。

命中附件驱动、跨表编排或专项诊断时，先读 `scenarios/INDEX.md`，再按命中 `SC-*.md` 里的专项风险和必要自检补充审视；不要把完整场景流程沉入本文件。

### 最小审视闭环（每次都做）

| 检查项 | 通过标准 | 常见风险 |
|--------|----------|----------|
| 主表选择 | 与用户确认、指标或路由一致 | 被字段名带偏、绕去专项表 |
| 业务口径 | 时间、状态、动作、客群明确 | 新装/到达/收入/积分口径混用 |
| 时间/分区 | 时间字段与口径一致，分区来自表文档 | 双重时间过滤误杀、假设分区 |
| 输出粒度 | 明细、汇总、分组维度互相匹配 | 明细字段混进汇总，导致重复行 |
| 可执行性 | 无中文占位符，无 `SELECT *`，别名和字段引用一致 | 参数不清、字段缺失、别名错位 |
| CTAS 形态 | 正式交付落 `tmp_*` / `ads_*`，不使用 `WITH` 或裸 `SELECT` | 中间结果不可验收、不可复跑 |

### 风险触发审视（命中才做）

| 触发条件 | 必查项 | 常见风险 |
|----------|--------|----------|
| 有补表或 JOIN | JOIN 键、补表粒度、去重、是否放大行数 | 一对多放大、补错机构层级 |
| 有状态 / 动作 / 参数码值 | 码值来源、是否可稳定复用 | 中文状态进 WHERE、猜码值 |
| 有附件 / 种子表 | 输入行数、命中数、输出行数、一对多原因 | 漏匹配、重复膨胀、顺序丢失 |
| 多月 / T+n / 历史快照 | 月份窗口、日/月表选择、过程表血缘 | 月份错位、日表月表混配 |
| 多指标同表汇总 | 各指标条件独立，公共 WHERE 是 OR 超集 | 把互斥条件错误 AND 合并 |
| 敏感字段 | 脱敏逻辑、空值和长度异常处理 | 身份证 / 客户敏感信息裸出 |
| 复杂多步 CTAS | 每步表名、上游、过滤、验收 SQL | 过程表不可追溯、少字段导致下游失败 |

用户可见输出只展开风险触发审视中真正命中的风险；无风险时不输出完整审计表。

## 选表与字段

- 主表必须由业务事实、标准指标或用户确认经验决定，不由字段名相似决定。
- 用户已明确主表时，不再绕去专项表，除非主表字段明确不足且用户确认。
- 任意产品入网量、新装量、到达量、规模统计默认主表为 069 全业务资料表；除非用户明确要求专项清单字段或专项指标口径，否则不要切到专项表。
- A 层表文档字段表是参考清单，不是完备清单；疑似缺字段时请用户或生产 `DESC` 核对。
- 每个输出字段都必须能回溯到“来源表 + 来源字段”。
- 落 SQL 前显式列出拟用生产表名；已知漂移：069 生产名 `dwm_yz_tb_comm_cm_all_final`，041 生产名 `dwm_yz_rpt_comm_ba_msdisc_final`。

## 口径与时间

- 069 入网类统计一般同时检查分区月和业务开通月：`par_month_id` 裁剪分区，`date_format(open_date,'yyyyMM')` 限定入网月份；不要只写分区月就当入网月。
- 069 日/月表选择（详见 `069_全业务资料表.md`）：查询区间全在近半年内 → 全程日表；否则若月表已覆盖**整段**所需账期 → 全程月表（不按半年前/半年后拆 UNION）；仅当月表缺月（常见当月未入月表）→ 月表 + 日表补缺口 `UNION ALL`。
- 主宽入网数标准口径：主表 069，`kd_desc='普通宽带'`、`is_new_user=1`、`date_format(open_date,'yyyyMM')=${month_id}`、`prod_type=40`。
- 主宽入网积分标准口径：对象筛选同主宽入网数；度量 `sum(jz_points)`；见 `M-BASIC-BB-002`。
- 动作类统计必须排除撤单作废：`COALESCE(subs_stat_reason, '-1') NOT IN ('1200','1300')`。
- **⚠️ 反模式：不要把订单表（040/041）的 `subs_stat_reason` 排除规则套到 069 资料表上。** 069 的 `is_new_user=1`、`is_cancel_user=1`、`is_wl_cancel_user=1` 等标签是已加工好的业务标签，无需叠加订单级排除条件。写过滤条件前先确认字段在当前主表中是否存在，不要靠记忆默认套用。
- “是否 X”默认输出标记列，不默认放进 WHERE；用户明确“只看 X”时才过滤。
- 时间字段要和业务口径一致：受理 `act_date`；订单表竣工 `subs_stat_date + subs_stat='301200'`；开通 / 069 入网 `open_date`；**主宽入网**（M-BASIC-BB-001/002）口语「按竣工」亦用 `open_date`，勿改 `subs_stat_date`；**拆机默认** `hist_create_date`（逻辑拆机 `is_cancel_user=1`）；用户明确物理拆机时用 `wl_cancel_subs_stat_date`。
- **历史某月快照 / 拆机前 N 月属性回溯**：069 必须用月表 `dwm_yz_tb_comm_cm_all_mon_final`，特性用月表 `tb_pre_cm_attr_all_mon`；禁止 069 日表 + 特性月表混配。
- 分区字段只按表文档 `partition_keys` 使用；分区是读哪些数据，业务时间字段是算哪些数据。
- 同一张表上输出多个条件指标时，优先用 `COUNT(DISTINCT CASE WHEN 条件 THEN serv_id END)` 或 `SUM(CASE WHEN 条件 THEN 字段 ELSE 0 END)`，避免多次扫表。
- 年度/月度指标打横输出必须明确起止账期；收入年度打横不要只写 `par_month_id >= ${start_month}`，应写 `par_month_id BETWEEN ${start_month} AND ${end_month}` 或等价闭区间。
- **117 实收 / 047 客户基本面·产数收入**（含附件清单、业务条件圈定、直查汇总）：完整规则见 `scenarios/SC-009_047117收入实收查询.md`；本文件不再重复公式与编排。
## 维表与 JOIN

- 销售品、产品等全省维表必须加城市过滤，销售品维表固定 `city_id=200`。
- 机构维表用 `org_id` 关联；分局/县分 `levs=3`，营服 `levs=4`。
- 每次 JOIN 的额外条件必须使用本次 JOIN 的别名，避免别名错位。
- 多对多或一对多补表必须说明行数放大风险；必要时先去重。
- 揽装人维表关联必须优先用 `staff_id`；`sales_code` 不唯一，禁止作为揽装人唯一 JOIN 或去重键，最多作为展示字段或辅助过滤。
- 补客户经理 11 开头 CRM 工号时，`sales_code` 只用于匹配 115 员工信息表的 `staff_code`；员工表须先按 `status_date desc, update_ts desc` 取最新记录，并过滤 `city_id='200'`、`staff_account like '11%'`。
- 地址 ID 关联统一转字符：069 `serv_addr_id` 是字符型，079 `id/addr_id_7` 是 `decimal(24,0)`；禁止默认 `cast(serv_addr_id as decimal(24,0))`，避免长地址 ID 或异常字符导致转换失败、漏数。

## 附件清单

- 用户附件提供序号、用户要求按原清单顺序回填，或输出需要与附件行对齐时，应保留附件序号并最终按序号输出。
- 附件无序号且无需回填顺序时，保留原始匹配键即可，例如号码、客户编码、工单号。
- 附件清单驱动的 SQL 必须核对输入行数、命中行数、输出行数；若 JOIN 后一对多膨胀，要说明膨胀原因和是否符合用户口径。
- 客户清单按客户名称兜底匹配时，必须提示客户名重名/多命中风险；有产权客户编码时优先用编码精确匹配。
- 一次性附件表名（如 `zone_gz_yz_*`）不要沉淀为运行时表索引，只抽象为“用户附件种子表”。

## 客户字段边界

- 号码/服务明细需求中，用户要号码对应的产权客户名、直销客户名、客户编码等字段时，优先使用 069 或当前事实主表自带客户字段。
- 号码/服务明细需求中，用户要号码对应客户年龄时，优先使用 069 的 `social_id/social_id_type` 计算，不绕去 108/109 客户表；固定写法：`case when length(a.social_id) = 18 and a.social_id_type = '1' then cast(from_unixtime(unix_timestamp(),'yyyy') as integer)-cast(substr(a.social_id,7,4) as integer) when length(a.social_id) = 15 and a.social_id_type = '1' then cast(from_unixtime(unix_timestamp(),'yyyy') as integer)-cast('19' || substr(a.social_id,7,2) as integer) else null end as age`。该口径按当前年份减出生年份，不做生日月日精算；不要输出身份证原文。
- 108/109 只用于客户实体映射或客户信息维护场景，例如签订/维护直销客户、通过产权客户找直销客户、按客户名称兜底补产权客户编码；不要为了普通号码明细客户字段绕去客户表。

## 机构解释

- 018 机构维表只负责把机构 ID 翻译成机构名称/层级字段；这个 ID 是直销客户归属、产权客户归属、号码归属、揽装归属还是其他归属，必须由来源表字段语义决定。
- 关联 018 时优先写成 `业务表机构ID = org.org_id`；只有需求和来源字段明确为分局/营服层级时，才附加 `levs=3/4` 等层级过滤。

## 揽装人 / 网点有效性

- 查有效网点下有效揽装人时，优先使用 113 揽装所属表；该表只包含“有效网点 + 有效揽装人”的组合。
- 判断为什么某网点没有号码或收入时，不要只看 113 缺记录；必须回查 112 网点维表和 111 揽装人维表，区分无效网点、有效网点但无揽装人、有效网点但揽装人无效。
- 历史账期必须用三张维表的 `_mon_final` 月表，并用 `par_month_id` 对齐业务账期。
- 身份证号输出时默认脱敏：保留前 6 位和后 4 位，中间用 `*`；同时处理空值和长度不足，避免 `repeat` 负数或空串异常。

## 码值

- 状态/动作字段禁止写中文术语：不要写 `subs_stat IN ('竣工')` 或 `action_type='新订购'`。
- 已收录码值从 `D_experience/dictionaries/{field}.md` 读取。
- 未收录码值必须问用户，不允许猜。
- 已知常用码值：`subs_stat='301200'` 竣工；`action_id IN (1292,6200)` 订购/销售品互换；`subs_stat_reason IN ('1200','1300')` 撤单/作废。

## SQL 形态

- 不使用 `SELECT *`。
- 过程表 `select a.*` 仅当需保留主表全字段且用户已确认时使用；裸 `SELECT *` 不允许。
- 临时表后续会用于过滤、JOIN 或输出的字段，必须在建临时表时保留；否则把过滤条件提前到建表语句中，避免后续引用不存在字段。
- 明细和汇总不要混在同一行级结果；需要两者时输出两段 SQL 或显式汇总结构。
- 不交付中文占位符 SQL；用参数占位或让用户补真实值。
- 分区裁剪必须来自表文档 `partition_keys`。
- 多对多 JOIN 要主动去重或说明行数风险。
- Hive 局部排序用 `SORT BY`；谨慎使用全局 `ORDER BY`。
- 统计用户数用 `COUNT(CASE WHEN 条件 THEN serv_id END)`。
- 多指标同表汇总、涉及服务数去重时，优先使用 `COUNT(DISTINCT CASE WHEN 条件 THEN serv_id END)`。
- 金额/积分用 `SUM(CASE WHEN 条件 THEN 字段 ELSE 0 END)`。
- 正式 SQL 默认不自动添加 `LIMIT`；仅当用户明确要求样例、预览、探查、Top-N，或自检 SQL 的样例检查时才使用。
- 大段成熟 SQL 脚本不要沉入 `RULES.md`；只在 `verified-cases/` 中按相似需求加载。

## SQL 编排模式

本节是 SQL 编排唯一规则权威。**正式交付 SQL 一律 CTAS 流水线，禁止 `WITH`。**

生成 SQL 前先判定单步或多步；无论简单或复杂，结果都必须落盘为 `tmp_*`（或约定的 `ads_*`）过程表，便于验收、改中间过滤和复跑下游。

| 模式 | 何时使用 | 交付 |
|------|----------|------|
| **单步 CTAS** | 单主表、无分列维表、无多段 `UNION`、逻辑一步可完成（含单表多指标 `CASE WHEN` 汇总） | 一步 `drop table if exists ... purge` + `create table tmp_* ... as select ...` + 验收 `select count(*)` |
| **多步 CTAS 流水线** | 满足任一：外部维表/配置导入；补表需单独验收；按年/按段不同 JOIN 键或过滤；多段 `UNION ALL`；种子/队列驱动；多月跟踪 / T+n；需求方可能改中间结果 | 每步 `drop purge` + `create table ... as` + 验收 `select count(*)`；附过程表血缘 |

**禁止（正式交付）**

- 任何 `WITH ... AS (...)` CTE 写法。
- 裸 `SELECT` 作为最终交付物（应改写为单步 CTAS 落结果表）。
- 用深层嵌套子查询代替过程表（中间结果不可单独 `select`、不可改过滤再跑）。

**非正式交付例外（允许简单 `SELECT`）**

- 自检 SQL、样例预览（如 `... LIMIT 10`）。
- `metrics/*.md` 等指标文件中的口径示例（agent 交付时必须改写为 CTAS）。

**反模式**

- 过程表缺少 `drop table if exists ... purge` 就重建。
- 号码级过程表内外都不 `sum(度量)`，导致同输出维度多行。

**CTAS 标准 DDL**

```sql
drop table if exists {table_name} purge;
create table {table_name}
row format delimited fields terminated by '\u0001'
stored as orc tblproperties('orc.compression'='snappy') as
select ...
```

**命名**：`tmp_{项目缩写}_{步骤语义}_{账期或年}` → `tmp_{...}_union` → `ads_{项目缩写}_{最终结果}`。

**汇总铁律**：号码/服务级过程表必须在某层 `group by 输出维度 + sum(度量)`（内层或外层至少一层）。

**标准案例**：`verified-cases/VC-20260520-001_按年科目清单匹配科目级税后收入.md`（按年分列维表 + UNION + 汇总）。

## 过程表血缘

多步 CTAS 交付前必须附过程表血缘。每步过程表要可单独验收，验收 SQL 通常为 `select count(*) from ...`；有关键去重或 JOIN 风险时，应补充 `count(distinct ...)`。单步 CTAS 至少附结果表验收 SQL。

| 步骤 | 表名 | 用途 | 上游 | 关键过滤 | 验收 SQL |
|------|------|------|------|----------|----------|
|  |  |  |  |  | `select count(*) from ...` |

## 审视清单

输出 SQL 后按两层检查：

1. **最小闭环**：主表、业务口径、时间/分区、输出粒度、可执行性、CTAS 形态。
2. **风险触发**：只有出现补表/JOIN、码值、附件、多月/T+n、多指标、敏感字段、复杂多步 CTAS 等场景时，才做对应专项检查。

用户可见部分默认不展示“通过项”。只在存在风险、假设、字段缺口、表名漂移、补表放大、月表覆盖不确定等会影响结果的事项时展开说明。

## 自检 SQL

静态审视查写法，自检 SQL 查运行期问题（空分区、JOIN 炸行、0 值、全 NULL）。交付正式 SQL 时默认附 2-4 条最必要的自检 SQL，由用户执行（或在有 SQL 执行通道时先跑）后回贴结果再确认交付。自检 SQL 要轻量、可直接跑；不要为了覆盖模板而堆满所有检查项。

| 自检项 | 目的 | 模板 | 期望 / 判读 |
|--------|------|------|-------------|
| 样例 | 看字段是否取对、有无 NULL/乱码 | `SELECT ... FROM 结果 LIMIT 10;` | 字段值符合业务直觉 |
| 量级 | 总量是否在合理区间、环比是否异常 | `SELECT COUNT(*), COUNT(DISTINCT serv_id) FROM 结果;` | 量级合理；服务级结果两者关系符合粒度 |
| 空分区 | 确认账期分区有数据 | `SELECT COUNT(*) FROM 主表 WHERE par_month_id=${账期};` | >0；为 0 说明分区未跑批或账期写错 |
| JOIN 炸行 | 补表是否放大行数 | 补表前主表行数 vs JOIN 后行数（或 `COUNT(*)` vs `COUNT(DISTINCT 主键)`） | 两者一致；不一致说明一对多需去重 |
| 关键字段质量 | 关键维度/度量是否大面积缺失 | `SELECT SUM(CASE WHEN 字段 IS NULL THEN 1 ELSE 0 END)/COUNT(*) FROM 结果;` | NULL 率在预期内 |

- 单步 CTAS：默认给结果表「样例 + 量级」；有分区不确定时加「空分区」；有补表/JOIN 时加「JOIN 炸行」。
- 多步 CTAS：默认给关键过程表 `count(*)` + 最终结果样例；只有存在字段质量风险时才补「关键字段质量」。
- 自检 SQL 由用户执行回贴，或在有 SQL 执行通道时先跑；结果异常先修口径/补表再交付。

## 风险输出模板

无风险时不要输出审计表。有风险或待确认时，使用短清单：

```text
风险 / 待确认：
- {风险项}：{影响}；{建议确认或处理方式}
```

需要展开审视过程时，最多输出命中的风险项，不列未命中的通过项。
