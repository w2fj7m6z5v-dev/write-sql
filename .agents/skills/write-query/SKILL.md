# write-query

把自然语言数据需求收口成可执行的 CDAP Hive SQL。`SKILL.md` 是唯一运行时流程权威；其它 md 只作为按需加载的资料。

核心原则：先澄清会影响 SQL 结果的关键缺口，再选相关 schema、确认业务口径、做字段映射和补表，最后生成完整 SQL 并做必要审视。用户入口和最终交付保持轻量；主表、字段、补表和审视细节只在低置信度、有歧义、存在风险或用户要求时展开。

## 启动策略

启动后只读本文件。不要自动读取全量表结构、全量指标或案例。

### 标准指标快速路径（命中时优先）

用户需求中出现可映射到 `METRIC_INDEX.md` 的标准指标名或同义表达（如「主宽入网量」「净增积分」）时，**在读 ROUTING / FIELD_BACKFILL 之前**按序加载：

1. `references/METRIC_INDEX.md` → 锁定指标 ID 与单指标文件路径。
2. `references/metrics/**/{metric_id}_*.md` → 锁定主表、过滤条件、时间字段。
3. 指标指向的主表 `references/tables/{id}_{表名}.md` → 锁定自带字段与表文档中的补表指引。

此路径下：**禁止**仅凭 `FIELD_BACKFILL.md` 的通用默认做补表决策；表文档与 `FIELD_BACKFILL` 冲突时以表文档为准（见「冲突优先级」）。

轻量澄清完成后，用**一次「方案确认」**收齐主表、口径（含指标 ID）、输出维度、补表清单；不要拆成「先主表、再补表、再等用户纠偏」多轮。

**命中 1 个标准指标**时走上方三步；**命中 ≥2 个**时改走下方「多指标编排路径」，单指标路径是其特例。

### 多指标编排路径（命中 ≥2 个标准指标时）

用户需求同时命中 `METRIC_INDEX.md` 中 **≥2 个**标准指标（或 1 个精确 + 若干待确认疑似）时，在读 `ROUTING` / `FIELD_BACKFILL` 之前执行：

**1. 指标发现（Metric Discovery）**

- 扫描 `METRIC_INDEX.md`，列出**全部**命中项（精确 / 同义 / 疑似），禁止只取第一个。
- 非标准指标度量（如「出账收入」「SUM(fee)」）记为**附加度量**，不冒充标准指标。
- 精确命中 **>4 个**时，先问用户是否全部输出（最多 1 问）。
- 仅「1 精确 + 疑似」时，只追问疑似项（最多 1 问）。

**2. 并行读取单指标文件**

对每个命中指标依次打开 `metric_file`，提取内部卡片：`metric_id`、主表、`par_month_id`/时间字段、粒度键、度量表达式、WHERE 切片。**禁止**把多个指标的 `WHERE` 直接 AND 合并。

**3. 兼容性分类（决定 SQL 形态）**

| 类型 | 条件 | SQL 策略 |
|------|------|----------|
| **A 同表同轴** | 主表相同；账期/时间口径一致；仅过滤切片不同 | 单步 CTAS + `COUNT`/`SUM` + `CASE WHEN` 条件聚合 |
| **B 同表异轴** | 主表相同，但时间字段或账期语义不一致 | 先澄清统一时间轴；可共用一次扫描时，各指标条件独立放入 `CASE WHEN`；时间范围、粒度或中间结果不同则拆多步 CTAS |
| **C 异表可对齐** | 主表不同，输出维度可对齐（分局/营服/月等） | CTAS 流水线 / `UNION ALL` / 按维度 JOIN |
| **D 不可对齐** | 主表、粒度或维度无法对齐 | 拆多段 SQL，或追问取舍 |

判定优先级：**A > B（澄清后可能变 A）> C > D**。命中 `verified-cases/INDEX.md` 中多指标案例（如 VC-20260427-001）时，优先对齐案例编排。

同表异轴多指标时，公共 `WHERE` 只能保留各指标共同需要的分区或范围条件；不同指标的业务时间条件必须留在各自的 `CASE WHEN`，不得直接 AND 合并。正式交付仍禁止 `WITH`；无法共用一次扫描时，使用多步 CTAS 表达独立底座。

**4. 主表锁定（多指标版）**

- 全部同主表 → 锁一张主表，读一次表文档；各指标条件只进各自 `CASE WHEN`。
- 异主表但 C 类 → 定驱动主表（通常行数最大或用户最关心的指标），其余子查询或 CTAS 再 JOIN。
- D 类 → 不强行一张 SQL；每段标注对应 `metric_id`。

**5. 一次方案确认（多指标模板）**

```text
方案确认（多指标）

命中指标：
- {metric_id} {metric_name} → 主表，条件摘要
- …

编排类型：A/B/C/D → {SQL 策略}

主表：{表名}（共享一次扫描 / 或多段说明）

共享维度：…
各指标度量：
- {列名} = COUNT(CASE WHEN <该指标完整条件> THEN serv_id END)  -- 来源 metric_id
附加度量：（非标准指标，如 SUM(fee)）

补表：（共享主表只补一次）
待确认：（仅 B/D 类或字段歧义）
```

**6. 字段映射、补表、SQL、审视**

- 字段映射与补表在共享主表上**只做一次**。
- SQL 规则：各指标条件**只进各自 `CASE WHEN`**；公共底座 `WHERE` 只能是各指标条件的 OR 超集，不能把互斥条件 AND 在一起。
- 需要去重时用 `COUNT(DISTINCT CASE WHEN … THEN serv_id END)`（见 `RULES.md`）。
- 存在包含关系（如 129+ ⊂ 主宽）时默认分列计数，不自动算占比，除非用户要。
- 审视追加：指标—条件一一对应、无错误合并 WHERE、超集过滤合理、附加度量口径已说明；默认不把完整审视表输出给用户。

**仅在这些情况追问用户**：B 类时间轴不一致；C 类驱动表/缺失维度填 0 还是 NULL；疑似命中；命中 >4 个指标需取舍。A 类同表同轴且条件清晰时**不要**为合并方式反复确认。

## 运行时禁读

下列文件**不要在运行时加载**，仅历史溯源用：

- `references/D_experience/_archive/*`：已归档的老经验层（`business_glossary`、`table_routing`、`cdap_global_rules`、`anti_patterns`、`field_backfill`）。内容已并入 `ROUTING.md` / `FIELD_BACKFILL.md` / `RULES.md`。
- `references/D_experience/lessons_learned.md`：仅复盘日志，不参与运行时决策。
- `references/MAINTENANCE.md`：知识库维护准则（维护者查阅），不参与运行时决策。
- `references/verified-cases/*.md`（除 `INDEX.md` 外）：仅当用户需求与 `INDEX.md` 中某案例「适用」列高度匹配时再打开命中案例，不要批量预读。
- `references/scenarios/*.md`（除 `INDEX.md` 外）：仅当用户需求与 `INDEX.md` 中某场景「适用」列高度匹配时再打开命中场景，不要批量预读。
- 任何 `*-workspace/`：评测/迭代工作区，不属于运行时资产。

## 路径选择（先判定）

收到用户需求后，先做「需求拆解与澄清」，再按缺口大小选择路径。不要因为信息略有缺失就自动进入完整 6 步；先问会改变 SQL 的关键问题。

| 触发条件 | 路径 | 处理 |
|----------|------|------|
| 信息足够，主表/口径/输出无明显歧义 | **直接方案确认** | 用 5 行以内说明主表、口径、输出、过滤、是否补表；用户确认后写 SQL |
| 只缺少少量关键条件 | **轻量澄清** | 只问 1-3 个会改变 SQL 的问题；用户回答后进入方案确认 |
| 主表、口径、字段语义或补表路径有明显歧义 | **标准确认** | 分阶段确认主表、口径、字段、补表；每阶段只展开必要信息 |
| 命中 ≥2 个标准指标 | **多指标编排** | 走「多指标编排路径」；A 类单步 CTAS 条件聚合，C/D 类多步 CTAS |
| 多主表 / 外部维表 / 按年或按段编排 / 种子/队列驱动 / 含 UNION / 多月跟踪 T+n / 需求方可能改中间结果 | **多步 CTAS** | 先确认编排方案，再按过程表流水线生成 SQL 和血缘 |
| 信息足够、逻辑一步可完成 | **单步 CTAS** | 一步 `drop/create tmp_*` 落最终结果表 |
| 用户明确说「直接给 SQL」「不要分步」且需求清楚 | **直接生成** | 仍用 CTAS 落盘（单步或多步）；头部注释标注主表、口径来源和默认假设；**禁止 `WITH`** |

所有路径都必须保留 SQL 生成后的最小审视闭环；轻量澄清不是省略审视。审视按风险触发，默认不向用户展开完整审计清单。

## 分步确认协议

确认输出按风险裁剪：用户入口短，内部检查按风险触发。用户修正前置结论时，回到对应步骤重做，不要把旧假设带入 SQL。

| 阶段 | 输出块 | 何时展开 | 必须包含 |
|------|--------|----------|----------|
| 1 需求拆解与澄清 | `需求澄清` | 每次需求第一步 | 查什么、查谁、时间、结果形式、业务限制；缺口最多追问 3 个 |
| 2 方案确认 | `方案确认` | 信息足够或轻量澄清完成后 | 主表、口径（命中标准指标时含指标 ID）、输出、过滤、字段映射、补表（含 JOIN 键与行数风险）、关键默认假设；**一轮收齐**，不拆成多轮碎片确认；**多指标**时用多指标模板，逐指标列条件与度量 |
| 3 主表选择 | `主表确认` | 主表低置信度或多表都合理时 | 拟用主表、生产表名、表文档、选表理由、必要备选表 |
| 4 口径确认 | `业务口径确认` | 标准指标/动作状态/时间口径会改变结果时 | 口径名称、技术条件、时间字段、码值、来源、待确认项 |
| 5 字段与补表确认 | `字段/补表确认` | 字段语义不明、主表缺字段或 JOIN 有风险时 | 需求字段、候选字段、补表、JOIN 键、粒度、行数风险 |
| 6 SQL 交付 | `SQL + 精简确认` | 最终交付 | SQL、口径简述、必要风险 / 待确认、少量自检 SQL；复杂多步或用户要求时另附过程表血缘 |

第一轮禁止输出完整主表确认表、字段映射表、补表表格或 SQL。只有低置信度、有歧义或用户要求展开时，才展示备选表、不选择的表、字段血缘和 JOIN 风险。

## 六步流程

### 1. 需求拆解与澄清

第一步必做，但只做轻量澄清。先用业务口语理解用户需求，再内部映射到 SQL 要素；不要要求用户提供表名、JOIN 键或字段名，除非用户已经用技术语言描述。

业务澄清 5 问：

- 查什么：数量、金额、积分、比例，还是明细。
- 查谁：宽带、移动、销售品、订单、收入、积分、某批号码 / `serv_id`。
- 按什么时间算：受理、竣工、开通、出账、拆机，还是账期快照。
- 怎么出结果：全量汇总、按维度分组，还是明细字段。
- 有什么限制：产品、客户、状态、区域、是否排除撤单作废、指定主表 / 口径。

第一轮只追问会改变 SQL 结果的缺口，最多 3 个问题；能按 CDAP 经验合理默认的内容，先写为“默认假设”，不要拿来追问。信息足够时，不追问，直接进入“方案确认”。

输出格式：

```text
我先按这个理解：
- 查什么：
- 查谁：
- 时间：
- 结果：
- 默认假设：

还差这几个会影响 SQL 的点：
1. ...
2. ...
```

内部 SQL 要素检查（不作为第一轮用户问题）：落 SQL 前确认 `SELECT`、`WHERE`、聚合 / `GROUP BY`、`JOIN` / 补表、输出控制是否齐全；只有缺口会改变结果时再问用户。

### 2. Schema Linking / 主表选择

优先级：用户确认 > 标准指标 > `ROUTING.md` > `TABLE_INDEX.md`（仅查 Hive 名/file_path）> 表文档。

规则：主表由业务事实决定；用户已指定主表则跳过泛化路由；生产表名漂移须让用户校对。命中标准指标时，必须先完成「标准指标快速路径」三步加载，再进入方案确认；不得仅读 `ROUTING.md` 就定主表。低置信度时输出「主表确认」块（见分步确认协议 §3），含拟用主表、最多 2 张备选表及不选表原因。

### 3. Business Semantics / 口径确认

命中以下词时必须先确认口径：新装、入网、发展量、订购、互换、竣工、到达、在网、出账、有效、活跃、拆机、销户、续约、降档、收入、积分、怎么算、匹配规则。

检索顺序：标准指标（`METRIC_INDEX` + 单指标文件）→ 主表附注 → `ROUTING.md` → verified case → 标注「未找到稳定口径」请用户确认。需展开时输出「业务口径确认」块（见分步确认协议 §4）。

**主宽入网指标路径**：用户说「主宽入网 / 主宽入网量 / 主宽入网数」时，先按 `METRIC_INDEX.md` 命中 `M-BASIC-BB-001` 或 `M-BASIC-BB-002`，再以对应单指标文件的完整技术口径为准。SKILL 只规定加载路径，不复制指标筛选条件；涉及“按竣工”时，按指标文件确认其技术时间字段，不要自行改用订单表 `subs_stat_date`。

### 4. Column Linking / 字段映射

只加载已确认主表的表文档。逐个输出：

`需求字段 -> 主表候选字段 -> 是否满足 -> 来源说明`

规则：

- 主表已有且语义明确：直接取。
- **069 主表**：用户说「揽装分局 / 揽装局向」→ `channel_subst_name`；「揽装营服」→ `channel_branch_name`（见 `069_全业务资料表.md` 揽装机构口语别名）；与划小分局/营服（`subst_name` / `branch_name`）不同，但**不必反复确认**。
- 主表有多个候选字段：列候选，让用户确认。
- 主表只有 ID 没有名称：进入补表。
- 表 md 未列字段但业务上可能存在：提示需用户或生产 `DESC` 核对。
- 主表缺字段时必须先标记为“缺口字段”并进入 Join Planning；生成 SQL 前必须确保该字段来自主表文档或明确补表结果，禁止把其它表的常见字段直接写成主表字段。
- 不要先猜“直销客户编码”“客户编码”“受理人”“协销人”对应哪个字段。

需展开时输出「字段映射确认」块（见分步确认协议 §5）。

### 5. Join Planning / 缺口补表

只有主表缺字段或只有 ID/编码缺名称时才补表。

补表资料读取顺序（**不可跳步**）：

1. 主表 `tables/*.md` 的字段说明与补表指引（如 069 固定补 113）。
2. 仍缺口的字段，再读 `FIELD_BACKFILL.md` 作通用兜底路由。
3. 打开具体补表 `tables/*.md`；命中专项时读 `SC-*.md`。

`FIELD_BACKFILL.md` 不得覆盖主表表文档已写明的补表路径；若通用规则与表文档冲突，**以表文档为准**，不要交给用户纠偏。

补表方案必须说明：

- 缺口字段。
- 补哪张表。
- JOIN 键。
- 补表粒度。
- 是否可能放大行数。
- 必要过滤，例如销售品维表 `city_id=200`、机构维表 `org_id + levs`。
- 命中专项场景时，`FIELD_BACKFILL.md` 仅作路由指针，完整步骤读 `SC-*.md`。

需展开时输出「补表确认」块（见分步确认协议 §5）。

### 6. SQL Generation + Review

用户确认方案或关键歧义已消除后再写 SQL。生成前读取 `references/RULES.md`，按其中「SQL 编排模式」判定**单步或多步 CTAS**；**正式交付禁止 `WITH` 和裸 `SELECT`**；SQL 写法、最小审视闭环、风险触发审视和自检 SQL 均以 `RULES.md` 为唯一规则权威。

执行要求：

- 所有正式交付 SQL 一律 CTAS 落盘：简单需求单步 `tmp_*` 结果表，复杂需求多步流水线。
- 若需求命中专项场景，按 `references/scenarios/INDEX.md` 只打开命中 `SC-*.md`，把场景风险和必要自检并入内部审视。
- 最终默认输出完整 SQL、简短口径说明、必要风险 / 待确认、少量自检 SQL；不要默认输出完整字段来源表、审计通过表或性能说明。
- 多步 CTAS 内部必须检查过程表血缘；用户可见血缘仅在复杂多步、存在风险或用户要求时附简表。单步 CTAS 至少给结果表验收 SQL。
- 自检结果异常时，回到对应阶段修正口径、字段或补表，不要带病交付。

## 输出格式

每个确认阶段使用短表格或短清单，最后交付默认精简为：

1. 完整 SQL（单步或多步 CTAS 脚本；**禁止 `WITH`**）。
2. 口径简述：主表、时间口径、关键过滤、必要补表。
3. 风险 / 待确认：只列真正会影响结果或执行的事项；没有则写“无额外风险”或省略。
4. 自检 SQL：默认 2-4 条最必要的验收 SQL；按风险选择，不凑满所有检查项。

默认不要输出完整字段来源表、审计通过项表、性能说明长段落或过程表血缘。仅在字段歧义、补表/JOIN 风险、复杂多步 CTAS、专项场景、用户要求展开时，再补充对应明细。

## 冲突优先级

1. 用户确认过的生产表名、字段和口径。
2. 单指标文件中的技术口径 SQL。
3. 表文档 frontmatter、字段说明和条件速查。
4. `ROUTING.md`、`FIELD_BACKFILL.md`、`RULES.md`。
5. `verified-cases/`。
6. 自由推理，必须标注为假设。

## 回填原则

维护者完整落点表见 `references/MAINTENANCE.md`（运行时禁读）。任务结束后只回填会复用、会影响正确性的经验：

- 新业务术语或选表经验：补 `references/ROUTING.md`（术语映射 / 主表路由表 / 术语→字段反查）。
- 新字段补表规则：补 `references/FIELD_BACKFILL.md`。
- 新硬规则或反模式：补 `references/RULES.md`（通用审计清单）；**专项链路只回填 `scenarios/`，不在 ROUTING/RULES/FIELD_BACKFILL 写长规则**。
- 新码值：补对应 `references/D_experience/dictionaries/{field}.md`。
- 新复杂专项链路、附件驱动流程或跨表诊断：优先补 `references/scenarios/SC-*.md`，并在 `references/scenarios/INDEX.md` 加一行；核心文件只保留短入口。
- 新案例模板：在 `references/verified-cases/` 加 `VC-YYYYMMDD-NNN_<语义>.md`，并在 `verified-cases/INDEX.md` 加一行（注明「适用 / 不适用」）。
- 只作为复盘的内容放 `references/D_experience/lessons_learned.md`，不要运行时加载。
- **不要**回填到 `references/D_experience/_archive/`：那是已归档的老 D 层，不参与运行时。
