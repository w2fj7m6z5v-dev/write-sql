# Anti-Patterns 反 pattern 库

> **用途**：SKILL 第二步选表 + 第六步写 SQL 时主动检查。每条反 pattern 含「症状—正确—回避」三段式。
>
> **回填规则**：每次踩坑后立刻补一条；编号按 AP-NNN 递增。

---

## 选表类

### AP-001 字段名诱导选专项表

- **症状**：用户说"按销售品取发展量"，agent 因为某专项产品清单（燃气卫士到达清单）里恰好有 `prod_offer_code/prod_offer_name` 字段名匹配，就选了该专项表。
- **正确**：先确认用户要的是"任意销售品"还是"某专项产品"。任意销售品走 041 优惠订单表 + dws_offer JOIN；专项产品才走对应专项清单。
- **回避**：选表三问（[table_routing.md](table_routing.md) 末尾）—— 业务范围 / 动作 vs 存量 / 字段需求。
- **首次发现**：2026-04-28（销售品发展量场景）

### AP-002 字段不在 md 字段表 ≠ 表里没有

- **症状**：041 md 字段表没列 `cust_name`，agent 假设没有此字段，绕路 JOIN 069 取脱敏的 `cust_name_tm`。
- **正确**：A 层 md 字段表是"参考清单"不是"完备清单"。怀疑表里有但 md 没列的字段时，向用户确认或用 `desc tableName` 查。
- **回避**：写 SQL 前列出"拟使用生产表名 + 关键字段"清单让用户校对。
- **首次发现**：2026-04-28

### AP-003 表 md 的 hive_name 当成生产现网名

- **症状**：069 md 写 `ads_yz_tb_comm_cm_all_final`，agent 直接拼到 SQL 里，但生产实际是 `dwm_yz_tb_comm_cm_all_final`（前缀漂移）。
- **正确**：A 层 md `hive_name` 仅供参考，落 SQL 前必须在回答中显式列出"我准备使用 X 库 Y 表"，让用户校对。
- **回避**：SKILL 第三步硬规则。
- **首次发现**：2026-04-28

## 状态/动作过滤类

### AP-004 状态/动作码值用术语而非码值

- **症状**：写 `subs_stat IN ('竣工','正常')` 或 `action_type='新订购'`。
- **正确**：状态字段全部是数字/字符串码值，如 `subs_stat='301200'`、`action_id IN (1292,6200)`。码值查 [dictionaries/](dictionaries/) 或问用户。
- **回避**：未在 dictionaries 找到的码值，**必须问用户**，不允许猜。
- **首次发现**：2026-04-28

### AP-005 把"是否X"标志当 WHERE 过滤

- **症状**：用户要看销售品发展量明细，希望保留是否竣工标志，agent 把 `subs_stat='301200'` 直接写进 WHERE。
- **正确**：业务上的"是否X"默认作为输出标记列：`CASE WHEN subs_stat='301200' THEN 1 ELSE 0 END AS is_jg`，**不进 WHERE**。是否过滤要用户明确说"只看竣工的"才进。
- **回避**：标志位字段（is_*、is_X 命名约定）默认输出，不默认过滤。
- **首次发现**：2026-04-28

### AP-006 漏掉撤单作废排除

- **症状**：统计销售品发展量时只过滤 `subs_stat_date` 落区间，不排除撤单作废，多算一批撤回的订单。
- **正确**：发展量 / 订购量 / 受理量 等"动作类"统计，必须加 `COALESCE(subs_stat_reason,'-1') NOT IN ('1200','1300')` 排除撤单作废（参考 [cdap_global_rules.md](cdap_global_rules.md)）。
- **回避**：动作类统计 = action_id 过滤 + subs_stat_reason 排除撤单作废 + 时间窗，缺一不可。
- **首次发现**：2026-04-28

## 维表 / JOIN 类

### AP-007 维表漏 city_id=200

- **症状**：JOIN `dws_offer` 没加 `city_id=200`，跨城重号销售品错配。
- **正确**：全省维表 `dws_offer` / `dws_product` 等必须 `city_id=200`。
- **回避**：维表 JOIN 检查 [cdap_global_rules.md](cdap_global_rules.md) 全局硬约束清单。
- **首次发现**：2026-04-28

### AP-008 机构维表 JOIN 字段错位

- **症状**：用 `salestaff_subst_id` 关联 `dwd_yz_dim_org.subst_id`（而 `subst_id` 在维表里是"机构所属县分标识"，不是机构本身）。
- **正确**：机构维表用 `org_id` 关联，并按 `levs=3` 分局 / `levs=4` 营服区分级别。
- **回避**：机构维表关联模板见 [cdap_global_rules.md](cdap_global_rules.md)。
- **首次发现**：2026-04-28

### AP-009 多次 JOIN ON 别名错位

- **症状**：
  ```sql
  LEFT JOIN dim b ON a.x=b.org_id AND b.levs=3
  LEFT JOIN dim c ON a.y=c.org_id AND b.levs=4   -- 笔误：应为 c.levs=4
  ```
  限定条件写在了上一个 JOIN 的别名上，恒为假，第二次 JOIN 永远不命中。
- **正确**：每次 JOIN 的额外限定条件必须用本次 JOIN 的别名。
- **回避**：写完 JOIN 后逐条 ON 检查别名一致性。
- **首次发现**：2026-04-28

## SQL 写法类

### AP-010 明细行混汇总（窗口函数滥用）

- **症状**：在明细列里用 `COUNT(1) OVER (PARTITION BY par_month_id)` 把月汇总塞进每行。
- **正确**：要明细就纯明细；要汇总用 GROUPING SETS 或 UNION ALL 单独输出汇总行。
- **回避**：禁止"明细 + 行级月汇总"混合输出。
- **首次发现**：2026-04-28

### AP-011 假设分区字段

- **症状**：未确认表是否按 `par_month_id` 分区，就在 WHERE 加 `par_month_id IN (...)`。
- **正确**：分区字段以表 md 的 `partition_keys` frontmatter 为准；未标注的不要加，否则可能误杀数据或 SQL 报错。
- **回避**：写分区裁剪前必须看表 md 的 `partition_keys`。
- **首次发现**：2026-04-28

### AP-012 双重时间过滤冗余/误杀

- **症状**：`par_month_id IN ('202509','202510') AND date_format(open_date,'yyyyMM') IN (...)`。月切片 `par_month_id` 与动作时间字段双重过滤，跨月数据可能被误杀。
- **正确**：分区只为裁剪 IO，动作过滤用业务时间字段；两者不混。如确有跨月需求，时间字段范围放宽到能覆盖。
- **回避**：分区是"读哪些"，时间字段是"算哪些"，职责分清。
- **首次发现**：2026-04-28

### AP-013 占位符 SQL 当成"完成"

- **症状**：在 IN 里写 `'销售品编码1','销售品编码2'` 中文占位，标榜 SQL 已完成。
- **正确**：要么参数化（`:offer_code_list`），要么向用户索要真实值后填入；不要交付占位符 SQL。
- **回避**：SQL 完成判定 = 可直接复制粘贴执行。
- **首次发现**：2026-04-28

## SQL 编排类

### AP-016 复杂需求默认 WITH，过程不可改

- **症状**：多账期、分列维表、多年 UNION、种子表打标等复杂需求，agent 输出多层 `WITH ... AS (...)`，需求方无法对中间结果单独 `select count(*)`、改过滤或改 JOIN 后再跑下游。
- **正确**：复杂场景默认 **CTAS 过程表流水线**（`drop purge` + `create table ... as` + 过程表血缘 + 每步验收）；标准模板见 `verified-cases/VC-20260520-001`。`WITH` 仅当用户明确不要建表或一次性探查且不迭代，并标注「无过程表，不可分段改」。
- **回避**：`SKILL.md` 第 6 步「SQL 编排模式」；`RULES.md`「SQL 编排模式」反模式清单。
- **首次发现**：2026-05-21

## 澄清类

### AP-014 第一轮澄清问得不全

- **症状**：只问了"业务概念"和"时间口径"，漏问"撤单/动作过滤、客群范围、是否X 是过滤还是标记"。
- **正确**：销售品/订单类需求的标准澄清清单（5 项）：
  1. 业务术语对应的 CDAP 概念
  2. 客群范围（公众/商企/政企/小微/全部）
  3. 时间口径（受理/竣工/出账/分区切片）
  4. 状态过滤（撤单作废排除？是否限定竣工？）
  5. 动作过滤（订购/变更/退订/互换）
- **回避**：参考 [SKILL.md](../../SKILL.md) 第一步澄清模板。
- **首次发现**：2026-04-28

### AP-015 主表还没盘字段，就先猜字段映射

- **症状**：用户说“直销客户编码 / 客户编码 / 协销人 / 受理人”，agent 没先盘主表字段，就直接把它们映射成 `cust_code`、`cust_nbr`、某个协销字段或 `staff_id`。
- **正确**：先锁主表，再列“需求字段 -> 主表候选字段”对照。主表有多个候选字段时先让用户拍板；主表没有时再补表。
- **回避**：每个输出字段在写 SQL 前都必须能回答两个问题：来源表是哪张，来源字段叫什么。
- **首次发现**：2026-05-01
